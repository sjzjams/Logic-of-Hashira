**UI 流畅度方案**
| 场景 | 当前风险 | 落地措施 |
|---|---|---|
| 首页肌肉图动画 | 每帧 `replaceAllMapped` SVG、构建 ripple、模糊滤镜 | 把 SVG 颜色变更从字符串层移到 painter/分层 overlay；静态 SVG 用 `RepaintBoundary`；pulse 独立成小 repaint 区 |
| 模糊/阴影 | `BackdropFilter` 和多层 blur 在动画区域内 | 非必要模糊改渐变/半透明；保留时包 `RepaintBoundary`，减少 repaint 范围 |
| 列表滑动 | `shrinkWrap ListView` 嵌在 `SingleChildScrollView` | 数据超过 20 条时改 `CustomScrollView + SliverList` 或单一 `ListView.builder` |
| 动画复杂度 | `_pulseController.repeat` 常驻首页 | 非当前可见或低电量模式停 pulse；路由切后台时暂停 ticker |
| build 重复执行 | `GoogleFonts.*` 和 mock 数据在 build 中重复创建 | 静态样式收敛到 theme/常量；mock fixture 提到文件级 const 或 repository |
| 耗时任务阻塞 | 未来真实数据解析、AI 消息处理、SVG 处理会进 UI isolate | 大 JSON、SVG 预处理、统计聚合用 `compute`/`Isolate.run` |
| Shader 卡顿 | 首次复杂绘制可能 shader jank | Profile 模式记录 SkSL warm-up；发布前禁用 debug 性能叠层/调试 shader 工具，保留真实 profile 数据 |

**架构优化**
当前规范写明“局部 `setState`、无全局 store”，这在原型阶段合理。但下一阶段要避免“引入全局状态后全屏刷新”。

建议分三步：

1. 状态分片：`HomeActivityState`、`WorkoutPlanState`、`SettingsState` 分开，不要一个全局 AppState 承载所有页面。Flutter 方案可选 `ValueNotifier`/`ChangeNotifier` 起步，数据跨 3 个以上页面再引 Riverpod/Bloc。
2. 数据层解耦：把 `HomeMockService`、Plan 的 `Map<String, dynamic>` 列表、Detail 的 exercises 迁到 `models/` + `repositories/fixtures/`。UI 只消费 typed model，减少 build 内对象创建和耦合。
3. 生命周期按需销毁：Tab 保活要分级。Home/Plan 可保活；Coach、Profile 可懒加载；AI provider 在 Coach dispose 或会话结束时释放/复用，避免低频模块长期占资源。
